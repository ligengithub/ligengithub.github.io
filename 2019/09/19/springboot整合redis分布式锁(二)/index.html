springboot 整合redis 实现分布式锁
Created 2019-09-19| Updated 2019-09-26

之前写过Zookeeper的分布式锁，再来复习一下使用场景，知道前因后果，才能更好的分析它的合理性和起的作用。

两种锁

  • 本地锁
    应用场景，服务只在单台机器上部署。。比如我一个线程在修改数据库的某条记录，为了放在其它线程修改同一条记录，可以创建一个本地锁，修改时候加锁
    ，修改完毕释放锁，因为锁是本地的，所以其他线程是知道你加锁还是没加。就可以满足。
  • 分布式锁 当你火了，你发现一台机器处理不过来了。你就寻思再部署一台机器。但是问题就来了机器A加锁的时候，，机器B并不知道机器A加锁，也会去操作。就会导致问题，怎么办，找一个中间人。
    像极了 你去网吧上网，问网管有没有机器，有机器你就买票上机，没机器你就等着，等到别人下机。 网管就相当于分布式锁。你们都是一个个请求。。再来看一下。没有分布式锁，，没有网管的情况下。
    抢机器，坑定得打架，打赢了赔钱，打输了进医院。都是不划算的。 所以分布式锁还是很重要的。

redis分布式锁原理

redis 创建一个Set，然后每个请求操作的时候，就往set里面插入一条固定数据（加锁）。操作完了之后再删除这条固定数据（释放锁）。这样就可以了。
每个请求来的时候都会尝试往这个Set中插入数据（加锁），而Set中是不允许重复数据的，所以注定只有一个请求可以拿到锁。
同时 Redis插入数据的时候可以设置数据的生命周期，防止死锁。就像你花了5块钱，只能上一个钟，10块钱就可以上两个钟，这个就是超时时间，时间到了，你就得下机，让别人来上机。
